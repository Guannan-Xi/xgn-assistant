# AutoMediaProducer 友好 UI 版

这个版本只保留一条清晰工作流：

**一本书 PDF → 本地解析全书 Markdown/文本 → 大模型生成 source_ranges → 按 source_ranges 切分每集 PDF（仅供本地解析）→ 逐集本地 PyMuPDF4LLM/pypdf 解析 → 脚本模型只接收本集 Markdown/文本，禁止 PDF 直传 → 台词润色 → 输出台词和绘图提示词 → 可选生成配图 → 自动后处理统一封面与片尾 → 生成拆分分集脚本、LRC、图片与分集封面/首页。**

旧工程里的备份加载、旧版 Markdown 章节流程、统计面板、旧版兼容路径等旁支都已删除。当前只保留核心工作流，并新增“先切分 PDF，再逐章/逐集本地解析；失败后改用 pypdf 文本提取，不上传 PDF”的省 token 解析策略。

## 推荐使用方式：打开图形界面

双击：

```text
run_gui.bat
```

或运行：

```powershell
python AutoMediaProducer.py
```

没有命令行参数时，程序会自动打开桌面 UI。

## 主要变化

- 各个文本阶段都可以单独设置 provider 和模型：大纲生成、分集提示词、脚本生成、台词润色。
- 大纲生成默认使用 `gemini / gemini-3.5-flash`。
- Gemini 3 系列完整保留在 Gemini 模型下拉框：`gemini-3.5-flash`、`gemini-3.1-pro-preview`、`gemini-3.1-pro`、`gemini-3-flash-preview`、`gemini-3-flash`、`gemini-3.1-flash-lite`、`gemini-3.1-flash-lite-preview`。
- Gemini 默认模型为 `gemini-3.5-flash`，Pro 选项保留为 `gemini-3.1-pro-preview`。
- OpenAI 文本模型保留 `gpt-5.5` 和 `gpt-5.5-pro`；新增豆包/火山方舟文本 provider，可填写推理接入点 ID（`ep-...`）或控制台支持直接调用的 Model ID。OpenAI 生图模型保留 `gpt-image-2`。
- 分集数量不再手动填写，由大模型根据整本书结构生成的大纲决定。
- 页码偏移固定为 `0`，不再做额外偏移。
- Gemini、OpenAI、豆包/火山方舟的 API Key 都可以在 UI 中统一填写和保存到项目根目录的本地 txt 文件。
- 界面有“开始”和“停止”。停止会在当前 API 请求结束后，于下一步开始前安全停止。
- 大纲生成提示词、分集脚本提示词生成器、脚本 JSON 约束、台词润色提示词都可以在 UI 里编辑，并保留默认值。
- 脚本生成后会进入“台词润色”步骤，再输出最终台词和绘图提示词。
- 生图完成后会自动以后处理方式统一生成封面与片尾：封面底图由大模型负责，文字、logo、slogan、蒙版和排版全部由程序统一完成。
- 每集自动输出四种封面规格，并固定编号：A1=3:4、A2=9:16、A01=B站 4:3、A02=B站 16:9；四种规格统一由同一张 A/C 母图裁切。
- 同时自动输出唯一 C 结尾页：C=9:16，只显示 logo、品牌名、slogan、关注引导语和分享引导语。
- PDF 默认优先使用本地 PyMuPDF4LLM 解析：`auto/pymupdf4llm` 会先把 PDF 转成高质量 Markdown。
- 脚本阶段现在是省 token 顺序：先切出本集 PDF → 本地解析本集小 PDF → 成功则只把本集 Markdown 发给脚本模型，不上传 PDF；失败后改用 pypdf 文本提取，不上传 PDF。
- 台词润色阶段不再重复解析或上传 PDF，只润色已生成的台词，进一步减少 token 消耗。
- 分集拆分阶段会实际接入业务流：每个拆分分集输出 `01_台词.lrc` 和 `02_脚本.json` 中的 `voiceover_lrc`；开头承接来自上一分集台词摘要，结尾预告来自下一分集台词摘要；跨章时会读取下一章/下一集第一段内容。承接摘要默认复用“台词润色”阶段的模型配置，例如 `doubao / ep-...`。
- 如果只想把已有脚本重新交给豆包润色，请把“起始步骤”改成 `polish`；程序会复用前面的脚本 JSON，但不会再跳过台词润色。
- 大纲阶段因为还没有 source_ranges，会处理全书一次来生成切分标签；后续分集阶段不会反复解析整本书。

## UI 使用顺序

1. 选择书籍 PDF。
2. 选择输出目录。
3. 在“流程参数”里设置 PDF 解析优先级。默认 `auto`：大纲阶段用于全书解析；脚本阶段会先切分 PDF，再只解析本集小 PDF。无需 GPU、无需 torch、无需模型下载。
4. 在“各阶段文本模型设置”里分别选择大纲生成、分集提示词、脚本生成、台词润色使用的模型。大纲默认是 `gemini / gemini-3.5-flash`；需要更强质量时，可把脚本生成或台词润色单独切到 `gemini-3.1-pro-preview`、OpenAI 的 `gpt-5.5-pro`，或豆包/火山方舟的 `ep-...` 接入点。
5. 填写 API Key。也可以在“API Key（可保存）”区域一次性填写 Gemini、OpenAI、豆包/火山方舟的 Key，并勾选“运行时保存 Key 到本地 txt”。
6. 第一次建议勾选“跳过生图，只输出绘图提示词”。
7. 需要调整风格时点击“编辑提示词”。
8. 点击“一键测试 dry-run”验证流程，确认没问题后点击“开始”。
9. 需要中断时点击“停止”。

## dry-run 测试

`dry-run` 不调用任何模型，不消耗额度，只用于验证：

- PDF 能否读取和切分；
- 输出目录结构是否正确；
- 台词文件和绘图提示词文件是否能正常生成；
- UI 日志是否正常显示。

## API Key

API Key 可以在 UI 中直接填，也可以留空，让程序自动读取：

- 环境变量：`GEMINI_API_KEY` / `GOOGLE_API_KEY` / `OPENAI_API_KEY` / `ARK_API_KEY` / `DOUBAO_API_KEY`
- 豆包/火山方舟模型名可填在各阶段模型框里，也可用环境变量：`ARK_ENDPOINT_ID` / `DOUBAO_ENDPOINT_ID` / `ARK_MODEL` / `DOUBAO_MODEL`；Base URL 默认 `https://ark.cn-beijing.volces.com/api/v3`，可用 `ARK_BASE_URL` / `DOUBAO_BASE_URL` 覆盖。
- 项目根目录文件：
  - `gemini_api_key.txt`
  - `google_api_key.txt`
  - `openai_api_key.txt`
  - `ark_api_key.txt`
  - `doubao_api_key.txt`

点击“保存 Key”“打开 Key 管理”或勾选“运行时保存 Key 到本地 txt”后，会保存到项目根目录的对应 key 文件。每个阶段只需要选择 provider，程序会自动读取对应 provider 的 Key。

## 本地 PyMuPDF4LLM 解析优先

新增策略：程序遇到需要给模型读取 PDF 的阶段时，默认先尝试本地 PyMuPDF4LLM。

- `auto`：默认模式，先使用 PyMuPDF4LLM 把 PDF 转成 Markdown。成功则把 Markdown 发给模型，不上传 PDF。
- `pymupdf4llm`：强制使用 PyMuPDF4LLM。
- `off` 已禁用：PDF 禁止直传给大模型；旧配置写 off 时会被强制改为 auto。

解析缓存会保存到输出目录下的 `_pymupdf4llm_cache/`，同一本 PDF 不会重复解析。

推荐设置：

```text
PDF 解析优先级：auto
本地解析器：pymupdf4llm
```

安装依赖：

```bash
pip install pymupdf4llm pymupdf
```

如果 PyMuPDF4LLM 未安装或解析失败，日志会显示原因，并自动进入 PDF 上传流程，不会直接中断。

## Windows 中文路径与网络断开修复

如果 PDF 或工程目录包含中文，例如 `著作解读`、`万历十五年.pdf`，程序在上传给 Gemini/OpenAI 前会自动复制一份临时英文文件名 `book.pdf` 用于接口调用。原 PDF 不会被修改。

如果本地 PyMuPDF4LLM 解析失败，程序会自动改用 pypdf 文本提取，保留 `[PDF Page N]` 页码标记；不会把原始 PDF 上传给任何文本模型。

`run_gui.bat` 和 `run_cli.bat` 也已设置 UTF-8 运行环境，减少 Windows 控制台中文编码问题。

## OpenAI 模型选项

- 文本模型：`gpt-5.5`、`gpt-5.5-pro`。
- 生图模型：`gpt-image-2`。
- OpenAI Responses API 调用默认不传 `temperature`，避免 `Unsupported parameter: temperature`。
- UI 的模型输入框仍允许手动输入，便于后续兼容新增模型名。


## A 系封面规格（固定）

- A1 = 3:4
- A2 = 9:16
- A01 = 4:3
- A02 = 16:9

程序会在每集 `06_封面与片尾/封面规格检查.json` 中输出校验结果；若定义不一致，会直接报错停止，避免生成错误规格。

A 系封面与 C 结尾底图强调“背景丰富、细节充足、避免整图蒙版”；后处理只保留局部文字框与边框，不再覆盖整张背景。
- A 系封面会保留完整关键信息：栏目名、钩子标题、书名、作者、期数/章节名、品牌 logo 与 slogan。
- C 末页只保留品牌信息：`知识慢炖` 与 `让经典不再高冷，让智慧人人可用`。
- 片头片尾整体风格参考经典历史解读封面：黑金+深红色调、古典人文氛围、金色装饰边框、背景丰富。



## 本版优化（省 token / A 与 C 共用母图）

- 每集只生成 **1 张 A/C 共用母图**：`A1_A与C共享母图`。
- A 系封面与 C 结尾页都从这同一张 9:16 母图裁切与后处理得到，不再为 C 单独生图。
- 这张共用母图会在提示词中要求先理解文稿语义，提取时代背景、人物、地点、事件、情绪、视觉意象和历史符号，同时保留顶部标题区、中部主视觉区、底部品牌区。
- `06_封面与片尾/` 下会额外输出 `A_C共享母图.png`，方便你核对。
- 为避免中文乱码，后处理字体优先使用微软雅黑/黑体/Noto CJK 等中文字体，并统一清理异常字符。


## 生图成本优化默认值

- OpenAI 生图默认使用 `quality=low`，可用环境变量 `AMP_IMAGE_QUALITY` 覆盖。
- A/C 共用母图默认走最低成本 `720x1280` 9:16 源图，可用 `AMP_SHARED_AC_IMAGE_SIZE` 覆盖；最终封面/片尾发布尺寸由本地后处理裁切、放大、锐化完成，不额外调用生图接口。
- 普通 B 系配图默认使用同样的最低成本 `720x1280`，可用 `AMP_B_IMAGE_SIZE` 覆盖。
- A1/A2/A01/A02 与 C 结尾页的最终发布尺寸仍由本地后处理裁切/缩放完成，不额外调用生图接口。


## 内置品牌 Logo

- 已内置并保存你提供的 `知识慢炖` logo：`automedia_core/assets/knowledge_slow_stew_logo.png`
- 程序会在每次生成 **C 系末页** 时自动调用这个 logo。
- 末页会优先使用 logo 的图形部分并裁切为**圆形**，再配合品牌名 `知识慢炖` 与 slogan `让经典不再高冷，让智慧人人可用`。
- 如需替换，只要覆盖同名文件即可。


## 结尾口播规则

- C 系末页视觉仍只显示：`知识慢炖` logo 与 slogan。
- C 对应台词会自动加入：
  - 下集内容提示；
  - 关注 `知识慢炖`；
  - 引导转发、分享给同样需要的人。
- 如果当前集不是最后一集，程序会读取下一集标题生成下集预告；如果已经是最后一集，则提示后续继续读完整本书。


## 逐句画面 / LRC 输出

- 脚本阶段会强制要求：**每一句台词都对应一幕画面**，单幕建议 **3~8 秒**。
- 程序会自动让 `image_prompts` 与 `voiceover` 尽量一一对应，并把对应台词锚定进同图号的画面 prompt。
- 每集会额外输出：`03_台词.lrc`
- LRC 采用 **30 秒步长时间戳**，方便你手动剪辑。
- 在 LRC 中，**画面编号单独成行**，例如：
  ```
  [00:30.00]B01
  [00:30.00]这是一句台词
  ```


## 编号职责（已锁定）

- A 系列只负责封面：
  - `A1` 是封面/片尾共用母图的脚本编号。
  - `A2 / A01 / A02` 只是后处理导出的规格文件名，不会出现在正文台词或绘图提示词里。
- B 系列才是正文内容画面：
  - `B01 / B02 / B03...` 与正文台词逐句对应。
  - 每一句正文台词都会对应描述其内容的一幕画面。
- C 只负责结尾：
  - C 会出现在台词与 LRC 中，作为结尾口播。
  - C 不单独生成 image_prompt，也不作为正文内容图。
  - C 末页由后处理复用 A1 共用母图生成，只显示 logo 与 slogan。


## v31 修正

- **B 系列才是正文内容图**，A 系列只做封面，C 系列只做结尾。
- **C 图不再混入正文内容图**；结尾只保留品牌卡逻辑。
- **C 图视觉只保留**：知识慢炖 logo、slogan、引导关注、引导转发/分享。
- **A 系封面只保留**：钩子标题（居中）、书名（居中）、作者、知识慢炖 logo、slogan。
- 后处理会**固定输出 4 张 A 系封面**与**4 张 C 系片尾**。


## v32 封面精简规则

A 系封面只保留以下元素：
- 标题（含第几集），居中排版
- 书名，居中排版
- 作者
- 知识慢炖 logo
- slogan

其他元素一律不显示。

封面标题会优先使用：
- `第几集｜凝练钩子标题`

例如：`第一期｜不上朝的皇帝，到底在跟谁赌气？`

## v33 本地安全排版优化

- A/C 系列所有文字均由 `automedia_core/postprocess_cards.py` 使用 Pillow 本地后处理完成。
- 大模型只负责生成无文字背景图，不再负责封面文字排版。
- A 系封面固定安全区：标题、书名、作者、logo/slogan 分区互不重叠。
- C 系片尾固定安全区：logo、slogan、关注、转发分享分区互不重叠。
- 本地排版会自动换行、自动缩字号、居中对齐，并用局部文字底板增强可读性。
- 不使用整图蒙版，不让文字溢出画面边界。


## v34 最终标准

  - 标题（含第几集）
  - 书名（自动补全为《书名》）
  - 作者
  - 知识慢炖 logo
  - slogan
- C 系片尾只保留：
  - 知识慢炖 logo

最后润色阶段只负责把表达改得通俗、清楚、顺口，不能删掉生成阶段已经覆盖的原文要点。

## 失败响应拦截

新版会拦截 `Response(id=...)`、`ResponseReasoningItem(...)`、`IncompleteDetails(reason='max_output_tokens')` 这类 SDK 响应对象。它们说明模型调用没有返回正文文本，不能被当作分集提示词、脚本、台词或绘图提示词继续使用。程序会停止或重建，不再把这类内容写进 B01。

- 商业 / 管理
- 心理学 / 行为科学
- 科学 / 技术

- 社科：问题 → 误解 → 案例 → 数据/实验 → 机制 → 政策启发
- 商业：场景痛点 → 模型 → 案例 → 方法步骤 → 失败条件
- 心理：日常困惑 → 心理机制 → 研究证据 → 误区 → 行动启发
- 科学：现象 → 概念 → 证据 → 模型 → 应用 → 限制
- 哲学：问题 → 概念 → 论证 → 反驳 → 现实意义
- 文学：人物处境 → 关系冲突 → 转折 → 意象 → 主题

### 3. 绘图提示词也按书籍类型切换

非历史书不会再默认使用宫殿、奏疏、皇帝、明代服饰、龙纹、黑金暗红历史风。

- 商业书：办公室、白板、流程图、团队讨论、产品路径
- 心理书：人物远景、日常物件、分岔路径、柔和光影
- 科学书：实验室、数据可视化、模型结构、自然现象
- 哲学书：书房、门廊、抽象光影、人物背影
- 文学书：电影感生活场景、人物关系、象征物、情绪空间
- 财经书：账本、图表、市场、城市远景、风险路径
- 健康书：干净诊室、生活方式场景、研究图形，不画病体特写

### 4. 可继续扩展单书配置

`config/book_profiles/通用书籍自适应.json` 是通用配置。

如果以后某本书需要特殊风格，可以在 `config/book_profiles/` 下新增同名 JSON，例如：

- `原则.json`
- `思考，快与慢.json`
- `活着.json`
- `枪炮、病菌与钢铁.json`

工程会优先使用通用规则，后续可逐本细化。

## 有没有更好的后处理库

当前工程继续使用 Pillow，优点是稳定、轻量、容易打包。
但如果你追求更强的视觉表现，推荐优先级如下：

1. **HTML/CSS + Playwright 截图**：最适合做爆款封面。排版自由度最高，做大字标题、描边、阴影、渐变、角标、标签都更容易。
2. **skia-python**：高质量矢量绘制和文字效果很好，性能和观感都比纯 Pillow 更强。
3. **Cairo / Pango**：中文排版更稳，适合复杂文本布局。
4. **Wand / ImageMagick**：做模糊、辉光、合成特效能力强，但部署和跨平台兼容稍重。

如果只选一个方向升级，我最推荐：**把封面后处理逐步迁移到 HTML/CSS + Playwright**。

## HTML/CSS + Playwright 封面后处理（新默认）

封面/片尾后处理已硬切到 **HTML/CSS + Playwright**：

- 默认渲染引擎：`playwright_html`
- 失败不回退：Playwright 或浏览器环境异常时直接报错
- 也就是说：A/C 系列只走网页排版方式渲染，不再静默落到旧渲染器。

### 这次迁移带来的好处

- 大标题、描边、阴影、玻璃面板、角标等视觉效果更容易继续强化；
- 复杂文本排版更灵活，后续做“爆款封面模板”更方便；
- 后续如果要做多套主题，只需换 HTML/CSS 模板或 JSON 配置，不必大量改 Python 绘图代码。

### 当前策略

- `render_cover()` / `render_end_card()` 会无条件调用 `automedia_core/postprocess_playwright.py`；
- `config/运行开关.json`、单集配置或全局规范中写入其它引擎值也不会生效；
- 如果 Playwright 渲染失败，流程直接失败，便于立刻发现环境或模板问题。

### 配置位置

当前唯一有效值：

- `playwright_html`：使用 HTML/CSS + Playwright 渲染

### 首次安装

```bash
pip install -r requirements.txt
python -m playwright install chromium
```

如果已经装过 Playwright，但没装浏览器，也要执行一次 `python -m playwright install chromium`。


## HTML 爆款封面模板系统 v2（本次升级）

这次把 **A1 / A2 / A01 / A02 / C** 的视觉模板进一步统一到了同一套 HTML/CSS 组件体系里，目标是更接近成熟短视频团队常用的封面生产方式。

### 设计原则

- **安全区优先**：不再依赖大量彼此挤压的绝对坐标块，而是改为栅格化内容区；
- **强钩子主标题**：本集名/下集预告始终是最强视觉中心；
- **品牌统一**：底部品牌栏统一成可复用组件，logo / 名称 / slogan 风格一致；
- **对脏底图更稳**：即使底图里残留旧字，也会通过整体压暗、模糊副底、内容壳层和独立标题卡尽量压住；
- **长文本可控**：统一走 JS 自动缩字号，极端情况下再做轻度横向压缩，减少重叠和溢出。

### 当前模板结构

- **A 系封面（竖版）**：书名 / 作者 → 章节信息 → 集数 → 主标题大卡 → 品牌栏
- **A 系封面（横版）**：左侧主信息壳层，右侧尽量留给主体画面
- **C 结尾页**：下集预告标题 → 下一集悬念卡 → 互动 CTA → 品牌栏

### 关键实现文件

- `automedia_core/postprocess_playwright.py`
  - 统一输出 A/C 系列 HTML 模板
  - 内置文本自适应 fitBox 逻辑
  - 使用 Playwright 直接截图成最终成品图

### 建议

后续如果还要继续打磨，可以优先从以下几个方向继续升级：

1. 做多套主题：例如“古籍金黑”“商业杂志”“纪实冷调”“知识卡片”等；
2. 给横版封面增加左右双模板：人物靠左 / 人物靠右自动切；
3. 为不同平台做差异模板：视频号、B站、小红书、抖音各自安全区不同；
4. 增加模板选择器，在 GUI 里一键切换主题。
